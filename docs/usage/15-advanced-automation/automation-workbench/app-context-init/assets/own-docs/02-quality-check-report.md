# 质量检查报告

## 目标

统一质量结论表达方式，让需求、实现、测试、发布、运行五个阶段的质量状态可以横向对齐、纵向追溯。

## 报告结构

- 背景：变更目标、范围、版本、责任人。
- 检查维度：功能正确性、稳定性、性能、安全、可观测性、回滚准备。
- 证据摘要：日志、测试结果、数据对账、代码审查、发布验证。
- 风险结论：按严重度分类。
- 处理建议：立即修复、带条件放行、延期发布、必须回滚。

## 结论等级

- `PASS`：核心风险已关闭，证据完整。
- `PASS_WITH_RISK`：可放行，但存在已知风险和前置监控要求。
- `BLOCKED`：前置条件不足，暂不能给出放行结论。
- `FAIL`：关键问题未关闭，不允许放行。

## 高质量报告的标准

- 结论必须绑定证据，而不是个人判断。
- 风险必须说明触发条件和影响面。
- 建议必须可执行，不能停留在“建议关注”。

## 归档要求

- 报告与版本号、任务号、发布时间窗关联。
- 报告必须保留最终结论和被关闭风险的闭环信息。
