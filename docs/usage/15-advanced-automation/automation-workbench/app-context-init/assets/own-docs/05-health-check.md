# 健康检查

## 目标

定义“服务可用”最低标准，避免把单一心跳成功误判为整体健康。

## 检查维度

- 进程：实例是否存活、重启是否异常。
- 接口：核心 API 是否可达、错误率是否异常。
- 依赖：数据库、缓存、消息、外部接口是否健康。
- 资源：CPU、内存、线程、连接池是否接近极限。
- 业务：关键交易路径是否成功。

## 分层定义

- `alive`：进程还活着。
- `ready`：可接流量。
- `healthy`：核心依赖正常，关键业务路径正常。
- `stable`：趋势平稳，无明显异常积压或错误升高。

## 常见误区

- 只看 200，不看延迟和错误模式。
- 只看单实例，不看整体分布。
- 只看基础设施，不看业务路径。

## 输出要求

- 给出健康等级，而不是二元结论。
- 标出最薄弱的链路点。
