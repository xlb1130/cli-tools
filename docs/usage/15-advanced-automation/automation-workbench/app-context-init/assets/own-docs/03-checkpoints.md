# 检查点

## 目标

把复杂交付流程拆成少量但关键的控制点，避免团队在“忙碌但无控制”的状态下推进。

## 阶段检查点

- 需求进入：目标、边界、非目标、约束是否清楚。
- 设计完成：关键方案、风险、兼容性、回滚是否明确。
- 开发完成：代码、配置、脚本、文档是否同步完成。
- 转测前：测试范围、测试数据、环境依赖是否准备好。
- 发布前：发布清单、回滚预案、监控口径是否明确。
- 发布后：日志、指标、数据、告警是否完成核验。
- 复盘时：问题、教训、规则更新是否闭环。

## 检查点设计原则

- 数量要少，但必须卡住高风险路径。
- 每个检查点都要有责任人和产出物。
- 每个检查点都要能回答“如果不通过会怎样”。

## 通过标准

- 能给出明确的通过、带条件通过或不通过。
- 不能用“基本没问题”代替检查结论。
