# Git 工作流

## 目标

用最小复杂度支撑多角色协作，保证变更可审阅、可回退、可追责。

## 分支策略

- 主干始终可发布。
- 特性开发使用短生命周期分支。
- 紧急修复应有清晰的 hotfix 路径，并回合主干。

## 提交规范

- 一次提交只解决一个逻辑问题。
- 提交信息要包含意图，而不只是动作。
- 不把格式化、重构、逻辑修复混在同一提交里。

## 合并要求

- 必须完成代码审查。
- 必须绑定需求、缺陷或任务上下文。
- 必须能说明为什么现在值得合并。

## 回滚原则

- 回滚优先于线上热修，前提是回滚成本低于修复成本。
- 回滚动作必须明确影响版本、影响范围和后续补救计划。

## 架构师视角关注点

- 分支策略是否降低了集成成本，而不是制造长期漂移。
- 是否存在“只会前进不会回退”的危险变更。
