# Common Scripts

## 目标

这一目录承载 `automation-workbench` 在多个 skill 之间共享的基础脚本能力。它们不是业务脚本，而是执行追踪、报告、标识生成和通用工具层。

## 设计原则

- 脚本必须保持小而稳，不承载复杂业务编排。
- 脚本能力要可复用，不与单一 skill 强绑定。
- 脚本应优先作为库能力存在，避免直接写死运行环境。
- 高风险动作不应放在公共脚本中默认执行。

## 文件职责

### `utils.py`

职责：

- 时间、路径、序列化等通用工具
- 公共小型辅助函数

适用场景：

- 其他脚本都可依赖

约束：

- 不放业务规则
- 不放网络副作用

### `report_uploader.py`

职责：

- 上报通用执行结果
- 统一上报载荷结构

适用场景：

- 任务完成后的统一结果上报
- 自动化运行结果归档

约束：

- 负责上报协议，不负责解释业务结果

### `task_reporter.py`

职责：

- 绑定 `task_id` 的任务结果上报
- 输出任务级状态摘要

适用场景：

- 单次任务执行收尾
- 批量任务中单任务状态记录

约束：

- 不直接做任务编排

### `token_collector.py`

职责：

- 采集 prompt/completion/total token 消耗
- 形成成本分析基础数据

适用场景：

- 模型调用后的成本统计
- 高消耗任务识别

约束：

- 只做采集和聚合，不做优化决策

### `self_evolution_reporter.py`

职责：

- 输出经验沉淀和复盘记录
- 形成自进化数据源

适用场景：

- 事故复盘
- 规则更新建议
- 高价值最佳实践沉淀

约束：

- 不直接改规则，只负责记录和上报

### `task_id_generator.py`

职责：

- 生成任务唯一标识
- 支撑跨脚本、跨阶段追踪

适用场景：

- 新任务初始化
- 批量任务拆分后的子任务标识

约束：

- ID 生成策略应稳定，不依赖外部系统状态

### `hooks/`

职责：

- 放置任务前后通用 hook
- 承载轻量、可复用的执行切面

适用场景：

- 初始化校验
- 任务结束摘要记录

约束：

- Hook 只做轻量切面逻辑
- 不在 hook 中隐藏高风险副作用

## 调用边界

- `app-context-init` 负责分发这些脚本，不负责实现复杂业务逻辑。
- 其他 skill 可以依赖这些脚本，但不应在自己的文档中重新定义脚本职责。
- 如需新增公共脚本，先判断它是否会被两个及以上 skill 复用。

## 新增脚本准入标准

- 是否具备跨 skill 复用价值
- 是否能以稳定输入输出形式存在
- 是否不会引入隐藏副作用
- 是否能被 README 中一句话说清楚职责
