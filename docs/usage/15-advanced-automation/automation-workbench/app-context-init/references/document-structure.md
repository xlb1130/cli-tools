# Document Structure

## 目标

统一 `app-context-init` 下文档的表达结构，使不同作者产出的文档在阅读成本、信息密度和执行性上保持一致。

## 推荐模板

### 治理类文档

适用对象：

- `own-docs/` 中的大多数文档

推荐结构：

1. `## 目标`
2. `## 核心原则` 或 `## 设计原则`
3. `## 关键规则`、`## 检查维度` 或 `## 风险控制`
4. `## 输出要求`、`## 结论要求` 或 `## 架构师关注点`

### 契约类文档

适用对象：

- `common-docs/` 中的大多数文档

推荐结构：

1. `## 目标`
2. `## 字段`、`## 状态模型`、`## 指标分层` 或 `## 基本规则`
3. `## 更新规则`、`## 处理策略` 或 `## 使用方式`
4. `## 输出要求` 或 `## 注意事项`

## 写作要求

- 先定义目标，再定义规则。
- 规则必须可执行，不写空泛口号。
- 能用列表表达就不用长段落堆砌。
- 结论类内容必须能指导动作，而不是只描述现象。

## 架构师视角要求

- 文档要能约束复杂系统，而不是只约束理想路径。
- 文档应优先暴露边界、失败模式和权衡。
- 对高风险动作，要给出升级条件和人工确认点。

## 不推荐的写法

- 只有背景，没有规则。
- 只有原则，没有执行条件。
- 只有命令，没有风险说明。
- 把临时经验写成永久规范。
