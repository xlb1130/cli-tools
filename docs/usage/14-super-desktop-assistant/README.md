# Super Desktop Assistant 产品与功能规划（草案）

本文面向产品规划与落地执行，目标是把“统一 AI 工具入口 + 可编排执行 + 可治理权限”具体化为可实施的桌面端方案。  
默认假设服务端具备类 Manus 的 Agent 编排能力，且支持 MCP、Skill 与云沙箱。

## 1. 产品定位

### 1.1 一句话定义

Super Desktop Assistant（简称 SDA）是一个桌面端“任务执行操作系统”：

- 用户只需要描述目标
- 系统自动选择并编排 CLI、桌面工具、MCP、Skill、云沙箱
- 全过程可审计、可回放、可治理

### 1.2 不做什么

- 不做纯聊天壳子
- 不替代每个专业工具的完整交互体验
- 不在 V0 阶段做开放生态市场

## 2. 目标用户与核心场景

### 2.1 用户分层

- 开发者与自动化工程师：高频使用 CLI 和代码工具
- 技术 PM 与数据/运营：需要“执行型助手”而非“问答型助手”
- 企业团队：要求权限、审计、预算与合规

### 2.2 高频任务场景

- 代码修复：复现、修复、测试、输出补丁建议
- 依赖升级：扫描、升级、回归测试、风险说明
- 发布前检查：lint/test/security/license 一键执行
- 日志排障：聚合异常、定位根因、给修复方案
- 文档同步：根据代码变更刷新文档与说明

## 3. 产品价值主张

### 3.1 统一入口

把多个 AI/CLI/桌面工具统一在一个任务界面中，减少切换成本。

### 3.2 统一编排

同一任务可跨本机、云沙箱、MCP 工具链协同执行。

### 3.3 统一治理

权限、审批、日志、成本统一纳管，可用于个人稳定使用与企业落地。

### 3.4 统一可复用资产

把“临时会话”沉淀为模板、工作流和 CTS 动态 CLI 命令。

## 4. 功能架构（桌面端视角）

### 4.1 Mission Control（首页）

核心组件：

- 任务输入框：自然语言 + Slash 命令（如 `/run`、`/plan`、`/sandbox`）
- 工具状态栏：显示已连接工具在线状态与版本
- 任务队列：进行中、待审批、失败、完成
- 最近资产：最近模板、命令、输出文件

关键行为：

- 用户输入目标后可直接启动“自动规划并执行”
- 可先选执行策略：本地、云沙箱、混合

### 4.2 Runbook（任务执行页）

核心组件：

- 步骤时间线：展示每一步使用了哪个执行器
- 实时日志：终端输出、结构化事件、错误聚合
- 文件变更：本地文件 diff 与产物列表
- 审批卡片：高风险动作人工确认
- 控制按钮：暂停、继续、重试、从步骤恢复、回滚

关键行为：

- 每个步骤有独立输入、输出、耗时、状态
- 失败步骤可选择“本地重试”或“切换沙箱重试”

### 4.3 Integrations（工具连接中心）

接入对象：

- CLI Connectors：Claude Code、QoderCLI、通用系统命令
- Desktop Connectors：Codex 类桌面工具、本机应用自动化
- MCP Connectors：工具发现、鉴权、调用测试
- Skill Connectors：Skill 版本管理、启停、回滚

关键行为：

- 首次连接时执行健康检查
- 展示每个连接器的权限范围与最近调用情况

### 4.4 CTS Builder（动态 CLI 工厂）

用户可配置：

- 命令名与别名（如 `proj.fix-tests`）
- 参数（必填、默认值、枚举、敏感字段）
- 执行链（本地步骤 + 沙箱步骤）
- 触发策略（手动触发/事件触发）
- 审批策略（如 push 前必须人工确认）
- 发布范围（个人/团队）

关键行为：

- 保存后即时可调用
- 每次发布生成版本号，支持回滚

## 5. 核心运行模型

### 5.1 任务状态机

建议统一状态：

- `Draft`：任务草稿
- `Planned`：已完成步骤拆解
- `Running`：执行中
- `WaitingApproval`：等待用户审批
- `Blocked`：环境不可用或权限不足
- `Failed`：执行失败
- `Succeeded`：执行成功
- `RolledBack`：已执行回滚

关键规则：

- 高风险动作必须进入 `WaitingApproval`
- 连续失败两次后自动建议切换云沙箱
- 任一步骤失败保留快照，支持从该步骤恢复

### 5.2 执行策略

- 本地优先：低风险、快速反馈任务
- 沙箱优先：高风险、长时、依赖复杂任务
- 混合执行：本地准备 + 沙箱重任务 + 本地收尾

### 5.3 可观测性与回放

- 每个任务绑定唯一 trace id
- 每步记录：输入、执行器、耗时、输出、变更、审批记录
- 支持任务回放与参数化重跑

## 6. Workspace 模型（桌面端关键差异化）

Workspace 是桌面端最核心安全边界，建议定义为“默认最小权限的执行范围”。

### 6.1 基本原则

- 用户首次使用必须选择本地 workspace 目录
- 默认只允许访问 workspace 及其子目录
- 超出范围访问一律拦截或弹窗申请授权
- 任务必须绑定 workspace，所有日志包含 workspace 信息

### 6.2 多 Workspace 支持

- 支持多个 workspace 并可快速切换
- 最近使用优先展示
- 每个 workspace 维护独立权限策略与历史记录

### 6.3 Trusted Workspace

对用户主动标记为可信的 workspace：

- 减少低风险重复确认
- 保留高风险动作审批（如删除、推送、部署）
- 仍保留审计轨迹，不跳过记录

### 6.4 Ephemeral Workspace（沙箱副本）

用于高风险和长任务：

- 在云沙箱克隆临时 workspace 执行
- 仅回传结果与可审阅变更，不直接污染本地
- 通过人工确认后再应用到本地工作区

### 6.5 Workspace 权限维度

- 路径权限：可访问目录白名单
- 动作权限：读、写、执行、网络访问
- 命令权限：允许命令、禁止命令、参数限制
- 环境权限：本地执行/沙箱执行/生产访问

### 6.6 Workspace 与工具注入规则

- CLI connector 执行时自动将 `cwd` 设为当前 workspace
- 桌面工具操作仅可引用当前 workspace 可见文件
- CTS 动态命令默认继承 workspace 权限策略

## 7. 桌面工具调用分层策略

桌面工具接入必须遵循优先级：`官方 API/CLI > 本地进程桥接 > GUI 自动化点击`。  
原因是三者在稳定性、可审计性、维护成本上有明显差异，应优先选择“合同明确、语义稳定”的方式。

### 7.1 方案 A：官方 API/CLI（首选）

适用条件：

- 工具提供正式 API、SDK、CLI 或稳定协议
- 可获取结构化输入输出（JSON/事件流）
- 版本兼容策略明确

交互实现：

- 在 `Integrations` 中配置连接器（endpoint、token、binary path）
- 发起任务时由 Orchestrator 调用标准接口：`start -> sendTask -> stream -> stop`
- 结构化事件进入统一事件总线（步骤状态、日志、补丁、审批请求）

能力边界：

- 支持最完整能力：计划生成、工具调用、补丁输出、状态查询
- 支持最强可控性：超时、重试、幂等、取消、并发限制

安全与审计：

- 所有调用绑定 workspace 与 trace id
- 审计粒度可到“每一步 API 调用与参数摘要”
- 可做策略化拦截（如禁用危险命令参数）

失败恢复：

- 接口级错误可区分：鉴权失败、配额限制、参数错误、执行超时
- 支持断点恢复与幂等重试
- 可快速回退到旧版本连接器

接入验收标准：

- 成功率、超时率、平均延迟达到阈值
- 至少覆盖 90% 主流程任务
- 输出事件能完整映射到平台标准事件模型

### 7.2 方案 B：本地进程桥接（次选）

适用条件：

- 没有官方 API，但可通过本地进程启动工具
- 能读取 stdout/stderr、退出码，或可附加 IPC 通道
- 工具行为相对稳定，命令参数可约束

交互实现：

- 桌面端启动受控子进程（固定 `cwd=workspace`）
- 通过 stdin/stdout、socket 或临时文件进行消息交换
- 由 Adapter 把非结构化日志归一成标准事件

能力边界：

- 可覆盖大量“CLI-first”工具，但语义稳定性弱于官方 API
- 对交互式命令需做额外封装（自动回答、超时中断、提示识别）

安全与审计：

- 必须启用命令 allowlist/denylist 与参数校验
- 进程级资源限制（CPU、内存、运行时长、网络策略）
- 记录完整命令、环境摘要、输出摘要、文件变更

失败恢复：

- 子进程卡死时可中断并收集最后输出
- 对可重复步骤支持“同参数重放”
- 进程崩溃后保留上下文快照，允许切换到沙箱继续执行

接入验收标准：

- 核心命令集合稳定通过率达到阈值
- 非预期交互提示可被识别并回退到人工确认
- 对崩溃、超时、权限拒绝有可观测错误码

### 7.3 方案 C：GUI 自动化点击（兜底）

适用条件：

- 无 API、无可控 CLI，仅有桌面图形界面
- 功能必须依赖 GUI 操作，且短期无法改造

交互实现：

- 使用 UI 自动化层进行窗口定位、元素识别、点击/输入
- 对关键步骤前后截图并做状态校验
- 每一步必须有“前置条件 + 结果校验 + 失败分支”

能力边界：

- 仅用于低频、非核心、可容忍波动的场景
- 对 UI 改版、分辨率、焦点抢占非常敏感

安全与审计：

- 高风险操作必须强制人工确认
- 强制保存操作录像/截图证据链
- 禁止在未锁定 workspace 的情况下执行文件相关动作

失败恢复：

- 元素定位失败后只能有限重试，超过阈值立即中止
- 中止后自动生成“人工接管建议”
- 不允许静默失败后继续执行后续步骤

接入验收标准：

- 明确声明为“Beta/Best effort”能力等级
- 每个动作都有可验证回执（截图或状态文本）
- 出错时可在 1 分钟内定位失败步骤

### 7.4 三种方案的统一抽象（必须）

无论底层如何接入，都必须输出统一协议，避免前端和编排器绑定某个工具实现。

统一任务接口（逻辑层）：

- `startSession(workspace, policy)`
- `submitTask(goal, context, constraints)`
- `streamEvents()`
- `requestApproval(action, reason, riskLevel)`
- `cancel()` / `terminate()`

统一事件模型（观测层）：

- `task.started` / `task.succeeded` / `task.failed`
- `step.started` / `step.finished`
- `tool.stdout` / `tool.stderr`
- `file.patch.proposed`
- `approval.required` / `approval.resolved`

### 7.5 Codex 修改项目（端到端交互示例）

用户目标：让 Codex 完成某个项目修改。

执行流程：

1. 用户在 `Mission Control` 输入目标，选择执行器为 `Codex`，并绑定 workspace
2. Orchestrator 先生成任务计划（读取代码、修改、测试、总结）
3. 按优先级尝试接入方式：
   - 优先使用 Codex 官方 API/CLI
   - 若不可用，降级到本地进程桥接
   - 再不可用时，仅对低风险步骤使用 GUI 自动化
4. Codex Adapter 连续输出步骤事件、日志、补丁建议
5. 触发高风险动作（如 `git push`）时进入 `WaitingApproval`
6. 用户批准后继续执行，完成后输出变更摘要、测试结果、回滚点

产品交互要求：

- 用户始终看得到“当前步骤、当前连接方式、风险级别、审批原因”
- 用户可一键切换执行策略（本地/沙箱/混合）并保留上下文
- 任务结束后可回放完整链路，支持复盘与合规审计

### 7.6 连接器能力等级（建议）

为避免“看起来都能接，实际体验差异巨大”，建议给每个桌面工具连接器打等级：

- `L1`：仅 GUI 自动化，可用性低，人工介入高
- `L2`：本地进程桥接，支持基本自动化与审计
- `L3`：官方 API/CLI 接入，支持完整治理与高稳定性

在 UI 中显式展示等级，并在任务规划阶段优先使用高等级连接器。

### 7.7 CLI 原生交互接入（以 qodercli 为例）

目标：在桌面端调用 qodercli 等 CLI 时，尽可能保留“用户直接在系统终端里运行”的原始体验，而不是把它简化成一次性非交互命令执行。

#### 7.7.1 交互保真目标

- 保留 TTY/TUI 界面显示（颜色、光标移动、进度条、分屏）
- 保留实时键盘交互（上下键、Tab、Ctrl+C、Ctrl+R 等）
- 保留历史与会话上下文（同一任务内多轮命令）
- 保留长任务流式输出和中途打断能力

#### 7.7.2 实现方式：PTY 会话桥接（推荐）

核心做法：

- 桌面端为每个 CLI 会话创建独立 PTY（pseudo terminal）
- 在 PTY 中启动 qodercli 进程，而不是普通 `subprocess pipe`
- 前端嵌入终端组件（如 xterm.js 类能力）渲染 ANSI 输出
- 键盘输入原样透传到 PTY，PTY 输出原样回传到 UI

为什么必须 PTY：

- 许多 CLI 通过 `isatty()` 判断是否进入交互模式
- 普通 stdout/stderr 管道会导致交互能力降级或丢失
- 只有 PTY 才能正确处理光标控制、窗口大小变化和终端信号

#### 7.7.3 桌面端交互流程

1. 用户在任务中选择“原生 CLI 模式（Interactive）”
2. 平台创建 `CliSession`，绑定 `workspace`、`profile`、`policy`
3. 创建 PTY 并以 `cwd=workspace` 启动 qodercli
4. UI 打开“原生终端面板”，双向转发输入输出
5. 用户可实时输入命令或让 Agent 注入命令
6. 遇到风险动作时挂起输入并弹审批卡片
7. 审批后恢复 PTY，会话继续

#### 7.7.4 必做能力点

- **按键透传**：支持控制键、功能键、组合键
- **窗口自适应**：前端 resize 需同步 PTY cols/rows
- **信号控制**：支持发送 SIGINT/SIGTERM（对应中断/终止）
- **编码处理**：统一 UTF-8，避免乱码导致命令误判
- **会话重连**：UI 断开后可重新附着到同一 PTY 会话

#### 7.7.5 原生模式与自动化模式并存

建议同一连接器支持两种模式：

- `interactive`：保留原生交互，适合人工协同与探索
- `managed`：结构化执行，适合自动化、重放、批量任务

调度策略：

- 用户手动接管时自动切到 `interactive`
- 执行模板化步骤时优先 `managed`
- 在任务内允许模式切换，但必须记录切换事件和原因

#### 7.7.6 安全与治理（原生交互也不能放开）

- 即使是 PTY 原生模式，也必须绑定 workspace 边界
- 命令输入在发送前执行策略检查（denylist/高风险模式匹配）
- 高风险命令（如 `rm -rf`、`git push`、部署命令）强制审批
- 全量记录会话审计：输入摘要、输出摘要、时间线、操作者

说明：为了避免泄露敏感信息，审计日志可默认脱敏（token、密码、私钥片段）。

#### 7.7.7 qodercli 接入建议（首版）

- 接入方式：优先官方 CLI（L3），落地用 PTY 运行时（交互保真）
- 默认命令模板：
  - `qodercli --version`（健康检查）
  - `qodercli doctor`（环境检查，如工具支持）
  - `qodercli`（进入交互主界面）
- 运行策略：
  - `cwd` 固定到 workspace
  - 注入受控环境变量（禁止继承不必要敏感变量）
  - 设置会话超时和空闲回收

#### 7.7.8 失败处理与回退

- PTY 启动失败：提示依赖缺失并回退到非交互模式
- 交互卡死：提供“发送 Ctrl+C”“强制终止”“转沙箱重跑”
- 输出异常：自动抓取最近 N 行日志与终端状态用于诊断
- 工具不兼容：将连接器降级标记为 `L2/L1` 并提示风险

#### 7.7.9 接入验收清单（DoD）

- 在桌面端运行 qodercli 的体验与系统终端一致度达到目标阈值
- 关键按键与终端 resize 全部通过回归测试
- 会话可中断、恢复、重连，不丢失核心上下文
- 高风险操作审批链路完整且可审计
- 至少覆盖 3 类任务：交互探索、脚本执行、长任务监控

### 7.8 成熟 CLI 工具单独接入（PTY 标准方案）

本节适用于已经“有完整原生终端交互体验”的成熟 CLI（如 qodercli、claude code、各类 DevOps CLI）。  
目标不是重写它们，而是在 SDA 中“无损承载”其交互能力，并补上治理、审计和编排。

#### 7.8.1 适用范围与接入原则

适用范围：

- CLI 本身依赖 TTY 交互（菜单、选择、回车确认、快捷键）
- CLI 已有稳定用户群，迁移成本高，不适合改造成 API-first
- 业务上需要保留原生体验，同时纳入平台治理

接入原则：

- 单工具单连接器（不要把多个成熟 CLI 混成一个黑盒连接器）
- 优先“透传原生交互”，最小化中间层改写
- “体验保真”与“策略治理”并行，不二选一

#### 7.8.2 架构组件

- `CliConnector`：工具元数据、健康检查、能力等级、策略绑定
- `CliSessionManager`：会话创建、附着、重连、回收
- `PtyRuntime`：真实 PTY 生命周期管理与 I/O 双向转发
- `TerminalView`：前端终端渲染层（ANSI、键盘、复制粘贴）
- `PolicyGate`：命令策略检查、风险动作审批、阻断执行
- `AuditRecorder`：会话事件、命令摘要、审批链、证据归档

#### 7.8.3 会话生命周期（状态机）

建议状态：

- `Init`：连接器初始化与环境校验
- `Ready`：PTY 已创建，等待命令
- `Active`：用户/Agent 正在交互
- `PausedForApproval`：触发风险动作，暂停输入透传
- `Detached`：前端断开但 PTY 保活
- `Terminated`：会话正常结束或强制终止
- `Error`：运行异常，等待恢复或回退

关键规则：

- 前端断线不等于会话结束（支持 `Detached -> Active` 重连）
- 审批时进入 `PausedForApproval`，审批完成后恢复到 `Active`
- 超时或策略触发时可强制进入 `Terminated`

#### 7.8.4 输入输出事件流（保留“选择 + 回车”交互）

典型时序（以 qodercli 选择方案为例）：

1. CLI 在 PTY 输出选项菜单（含光标控制与高亮）
2. `TerminalView` 渲染到用户界面
3. 用户按方向键或数字键进行选择
4. 按键事件原样写入 PTY（不改写按键语义）
5. 用户按回车，PTY 将 `\r`/`\n` 发送给 CLI
6. CLI 接收确认后进入下一步并持续输出
7. 平台同步记录“输入事件摘要 + 步骤进展”

实现要求：

- 支持 raw key 透传（包括控制键、功能键、组合键）
- 支持 bracketed paste，避免多行粘贴被错误拆分
- 必须同步 PTY `cols/rows`，确保 TUI 布局不乱

#### 7.8.5 审批拦截与恢复机制

在 PTY 模式下，审批不能通过“后处理日志”实现，必须做实时门控：

- 检测到高风险命令模式后，立即暂停输入流
- 进入 `PausedForApproval` 并弹审批卡片
- 审批通过：恢复输入流，继续原会话
- 审批拒绝：向 PTY 发送中断信号（如 Ctrl+C）并记录结果

拦截建议对象：

- destructive：`rm -rf`、批量覆盖、权限改写
- outbound：`git push`、远端发布、生产部署
- privileged：sudo/root 相关命令

#### 7.8.6 与 Workspace 策略联动

- 创建 PTY 前先做 workspace 绑定校验
- 强制设置 `cwd=workspace`，禁止越界路径执行
- 环境变量注入采用 allowlist（默认不继承全部系统 env）
- 对文件访问越界、网络越权动作进行即时阻断

#### 7.8.7 失败恢复与运维诊断

失败场景与处理：

- PTY 创建失败：提示依赖问题，回退到 managed 模式
- CLI 无响应：提供“发送 Ctrl+C”“发送 Ctrl+Z”“强制终止”
- 前端刷新或断线：支持会话重连与日志续看
- 输出乱码：自动校验 locale/encoding，必要时降级为纯文本模式

诊断证据建议保留：

- 会话元数据（PID、启动参数、cwd、connector version）
- 最近 N 行终端输出
- 关键按键事件摘要（脱敏）
- 策略命中与审批历史

#### 7.8.8 首版实施边界（建议）

- 先支持 1-2 个成熟 CLI（qodercli 必选）
- 先做单会话稳定，再做多会话并发
- 先保证交互保真与审计，再做复杂自动编排
- GUI 自动化仅作为该 CLI 的兜底，不作为主路径

#### 7.8.9 验收标准（PTY 专项）

- “选择 + 回车确认”交互成功率达到目标阈值
- 终端 resize 后 TUI 不错位
- 会话断开重连成功且上下文连续
- 高风险命令拦截、审批、恢复链路完整
- 与系统终端体验一致性达到可接受阈值

## 13. 技术选型与分层边界（方案 A）

本节明确采用方案 A，并定义每层职责，确保“全局编排可靠”与“单任务推理灵活”同时成立。

### 13.1 推荐技术栈

- 桌面端：`Electron + TypeScript`
- 任务编排主干：`Temporal + TypeScript workers`
- 模型访问层：`LLM 统一调用网关（OpenAI / Anthropic / 其他）`
- Agent 推理层（可选）：`LangGraph JS`（仅用于复杂推理节点）
- 执行层：`PTY runtime + MCP gateway + Skill runtime`

网关实现建议（优先）：

- `LiteLLM` 作为多模型统一入口（路由、fallback、限流、成本统计）
- 在 LiteLLM 前保留业务网关层，封装产品域策略（权限、审计、模型白名单）

### 13.2 为什么 Temporal 管全局、LangGraph JS 只管单任务推理

核心原则：

- Orchestrator 负责“做对”（可靠执行、可恢复、可审计）
- Agent 负责“想好”（规划、推理、工具选择建议）

原因：

- 全局流程需要强确定性与耐久状态（审批等待、长任务恢复、重试补偿）
- LLM 推理天然非确定，适合作为节点能力，不适合作为全局状态机
- 多执行器协作需要统一失败语义与统一审计链，工作流引擎更适配
- 未来替换 Agent 框架时，主干编排不受影响，架构更可演进

### 13.3 分层职责（按运行链路）

`Electron + TypeScript`（桌面层）：

- 用户交互、任务配置、Runbook 展示、审批入口
- 管理本地会话附着、终端渲染、文件变更可视化
- 不承载核心编排决策，不在本地维护全局任务状态机

`Temporal + TypeScript workers`（编排层）：

- 任务 DAG 调度、节点依赖、并发控制、超时与重试
- 人工审批等待（durable wait）与恢复
- 局部重跑、回滚点管理、失败补偿
- 统一生成任务事件流与审计事件

`LLM 统一调用网关`（模型访问层）：

- 统一模型路由、凭据管理、限流、成本统计
- 统一请求/响应格式与错误码
- 支持多供应商回退策略与模型策略配置
- 推荐通过 LiteLLM 聚合多厂商 SDK，减少业务层直连成本
- 对外暴露平台统一模型接口，屏蔽底层厂商差异

`LangGraph JS`（节点推理层，可选）：

- 仅用于“复杂推理节点”：任务拆解建议、工具候选排序、上下文摘要
- 不持有任务全局真相状态，不直接管理跨节点恢复
- 推理输出以结构化结果返回 Temporal 节点

`PTY/MCP/Skill 执行层`：

- PTY：成熟 CLI 原生交互（选择、回车、快捷键、重连）
- MCP gateway：工具发现、调用、鉴权、能力目录
- Skill runtime：Skill 版本、依赖、执行与结果标准化

### 13.4 单任务执行数据流（简化）

1. 桌面端提交任务配置（目标、workspace、执行策略）
2. Temporal 创建任务图并调度节点
3. 节点若需要复杂推理，调用 LangGraph JS 子图
4. 节点执行通过 PTY/MCP/Skill runtime 落地
5. 高风险动作进入审批等待，用户在桌面端确认
6. Temporal 恢复任务并继续，直至完成或可恢复失败
7. 结果回传桌面端 Runbook，形成可回放审计记录

### 13.5 节点边界建议（什么该放哪层）

应该放 Temporal：

- 审批等待与超时
- 跨执行器切换与回退
- 重试/补偿/回滚
- 成本与策略门控

可以放 LangGraph JS：

- 代码修复方案候选生成
- 工具调用顺序建议
- 多来源信息汇总与摘要

不建议放 LangGraph JS：

- 全任务生命周期状态机
- 审批状态持久化
- 长时间任务恢复逻辑

### 13.6 V0 实施建议（按工程优先级）

- 先落地 Temporal 主链路（任务创建、节点调度、审批等待、失败重试）
- 先用 LiteLLM 接入 1-2 个主力模型供应商，再逐步扩到多模型路由
- LangGraph JS 只接一个“复杂推理节点”作为试点
- 执行层优先：qodercli 的 PTY + 一个 MCP + 一个 Skill runtime
- 在 Runbook 中先跑通“节点级状态 + 审批 + 局部重跑”闭环

### 13.7 全局 Plan 归属与生命周期（Temporal 作为执行真相）

结论：

- 全局 Plan 的“执行真相”由 Temporal 持有
- LLM/LangGraph 只负责产出候选计划，不直接驱动全局状态推进

#### 13.7.1 三层 Plan 模型

- `Draft Plan`（候选计划）
  - 来源：LLM/LangGraph 推理结果
  - 特点：可变、可多版本、可比较
- `Validated Plan`（已校验计划）
  - 来源：策略引擎校验 + 用户确认
  - 特点：满足 workspace/权限/风险约束
- `Execution Plan`（执行计划）
  - 来源：Temporal workflow 接收并固化
  - 特点：成为运行时唯一依据（single source of truth）

#### 13.7.2 计划推进状态（建议）

- `PlanDrafted`：候选计划生成
- `PlanValidated`：策略与依赖校验通过
- `PlanApproved`：用户确认通过（如需要）
- `PlanCommitted`：写入 Temporal，进入可执行态
- `PlanAmended`：执行中修订（受控变更）
- `PlanClosed`：任务结束（成功/失败/回滚）

#### 13.7.3 执行中改计划规则（必须受控）

- 允许“局部改计划”，禁止无记录整体替换
- 任何改动都要生成 `plan_revision` 并记录变更原因
- 高风险改动（新增生产动作、扩大权限范围）必须重新审批
- 仅允许在 Temporal 中提交 plan patch，客户端不直接改执行真相

#### 13.7.4 失败恢复与 Plan 一致性

- 恢复时以 Temporal 历史事件重建当前计划视图
- 已完成节点默认冻结，避免重复执行
- 仅失败节点及其下游允许重排或重跑
- 若执行器切换（如 qodercli -> codex），必须记录为新 revision

#### 13.7.5 Runbook 展示要求（用户可见）

- 显示当前 `plan_revision` 与生效时间
- 显示“原计划 vs 当前计划”差异
- 标注每个步骤的来源：初始计划/运行时修订
- 提供“为何改计划”的解释字段（来自策略或人工操作）

#### 13.7.6 最小实现（V0）

- 支持 `Draft -> Validated -> Committed` 三段流转
- 支持单次运行时局部修订（1 次 patch）
- 支持在 Runbook 查看 revision 与差异
- 支持失败后从失败节点续跑，不全量重来
