# Super Desktop Assistant 界面交互设计（详细版）

本文聚焦桌面客户端的信息架构与交互设计，目标是把多能力系统（Skill、MCP、Workspace、原生客户端操作、成熟 CLI、CTS 自定义 CLI）清晰分区，避免用户在一个界面中“功能混杂、心智负担过高”。

## 1. 设计目标与边界

### 1.1 设计目标

- 把“任务执行”作为第一入口，能力接入作为第二入口
- 把“执行操作”与“能力配置”分离，避免误触和认知切换
- 把高风险动作与高频动作分层展示，提升安全与效率
- 让新手 10 分钟内完成首个任务，让高级用户可深度编排

### 1.2 设计边界

- 本文只定义桌面客户端交互，不展开后端接口细节
- 默认服务端具备任务编排、审计与策略能力
- 默认已支持 Workspace 权限边界与审批机制

## 2. 信息架构（IA）

客户端采用“1 个主工作台 + 5 个功能域 + 1 个全局面板”的结构。

## 2.1 顶层导航

- `Mission`：任务发起与执行（默认首页）
- `Runbook`：任务详情、步骤、日志、审批、回放
- `Integrations`：外部能力接入（Skill/MCP/客户端/CLI）
- `CLI Studio`：CTS 自定义 CLI 设计、发布与版本管理
- `Workspace`：工作空间与权限边界管理
- `Governance`：策略、审计、预算（团队版）

全局固定面板：

- `Command Bar`（Cmd/Ctrl + K）：全局搜索命令、资源、任务
- `Notification Center`：审批请求、失败告警、任务完成通知

## 2.2 功能域与职责区分（关键）

- `Mission` 只管“做事”：输入目标、选执行器、看进度
- `Integrations` 只管“接能力”：连接、鉴权、健康检查、能力等级
- `Workspace` 只管“边界”：路径范围、命令策略、权限授权
- `CLI Studio` 只管“造能力”：自定义 CTS 命令与工作流封装
- `Runbook` 只管“可观测与可恢复”：步骤、日志、审计、回放

禁止混放原则：

- 不在 `Mission` 页面做连接器配置
- 不在 `Integrations` 页面直接执行生产任务
- 不在 `Workspace` 页面执行命令，仅允许测试与校验

## 3. 核心对象模型（用于界面一致性）

- `Task`：用户目标的执行实例
- `Session`：某个执行器的运行会话（如 PTY CLI 会话）
- `Connector`：某种能力接入实例（MCP、Skill、CLI、客户端）
- `Workspace`：本地执行边界
- `Policy`：权限、审批、预算、风险规则
- `Template`：可复用任务模板
- `CTS Command`：用户发布的自定义命令

每个页面必须显式显示其主对象，减少用户“当前在编辑什么”的困惑。

## 4. 页面详细交互

### 4.1 Mission（任务首页）

#### 页面目标

让用户快速发起任务，并明确“使用什么能力、在哪执行、风险如何控制”。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| Top Nav: Mission | Runbook | Integrations | CLI Studio | Workspace | Governance |
+----------------------------------------------------------------------------------+
| Command Bar: [ 输入任务目标... ]                         [仅规划] [运行任务]      |
+----------------------------------------------------------------------------------+
| Recent / Templates              | Plan Preview                                  |
| - 最近任务 A                    | Step 1: 解析目标                               |
| - 最近任务 B                    | Step 2: 选择执行器                             |
| - 模板: 修复测试                 | Step 3: 执行与审批点预估                        |
| - 模板: 依赖升级                 | ...                                            |
+---------------------------------+------------------------------------------------+
| Execution Config                | Risk & Guard                                   |
| Executor: [Auto v]              | 风险等级: Medium                               |
| Workspace: [/proj/app v]        | 预计审批: git push, rm                         |
| Mode: [Local / Sandbox / Mix]   | 策略: Workspace Guard 已启用                    |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- 顶部：任务输入框（自然语言 + `/` 命令）
- 左侧：最近任务、收藏模板
- 中间：任务规划预览卡（步骤草案）
- 右侧：执行配置卡（执行器、workspace、模式、风险级别）
- 底部：快速动作（运行、仅规划、沙箱运行、保存模板）

#### 关键交互

- 输入任务后自动触发“规划预览”，不立即执行
- 用户必须确认三个要素后才可运行：
  - 执行器（Auto/Codex/qodercli/其他）
  - Workspace
  - 执行策略（本地/沙箱/混合）
- 如果任务涉及高风险操作，运行按钮旁显示风险标记与预计审批点

#### 状态设计

- 空态：展示“从模板开始”“连接工具”引导
- 规划中：显示步骤 skeleton 与预计耗时
- 规划失败：展示缺失能力（如未连接 MCP、未选择 workspace）

### 4.2 Runbook（任务执行页）

#### 页面目标

让用户看到“发生了什么、为什么卡住、下一步能做什么”。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| Runbook: Task #T-2026-0419 [Running] [暂停] [重试] [切沙箱重试] [从此步恢复]      |
+----------------------------------------------------------------------------------+
| Step Timeline            | Live Output / Diff                     | Approval   |
| 1. 读取项目结构   Done    | [stdout] ...                            | 待审批项    |
| 2. 修改依赖      Running  | [stderr] ...                            | - git push |
| 3. 运行测试      Pending  | --------------------------------------  | [批准][拒绝]|
| 4. 生成总结      Pending  | Files Changed:                           | 风险原因    |
|                          | - pyproject.toml                         | 影响范围    |
|                          | - tests/test_x.py                        | 审批历史    |
+--------------------------+-----------------------------------------+------------+
| Footer: TraceId=... | Executor=qodercli(PTY) | Workspace=/proj/app | Cost=...   |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- 左栏：步骤时间线（可跳转）
- 主区：当前步骤实时日志 + 文件变更 + 结构化事件
- 右栏：审批卡片、策略命中、资源消耗（token/沙箱时长）
- 顶部工具栏：暂停、继续、重试、从此步恢复、切换执行模式

#### 关键交互

- 任何失败步骤都提供三个按钮：`重试`、`切沙箱重试`、`人工接管`
- 命中审批时自动进入 `PausedForApproval`
- 批准或拒绝后，页面记录审批人、时间、理由
- 支持“回放此任务”并允许替换输入参数

#### 状态设计

- Running：实时流刷新
- WaitingApproval：日志区域半透明，输入受限，审批面板高亮
- Blocked：给出阻塞原因与修复建议（权限、连接器、配额）

### 4.3 Integrations（能力接入中心）

#### 页面目标

把能力接入统一管理，帮助用户理解“我已经具备哪些能力、质量如何、可否用于自动执行”。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| Integrations                                                                     |
+----------------------------------------------------------------------------------+
| Tabs: [Skills] [MCP] [Desktop Clients] [Mature CLI]                             |
+----------------------------------------------------------------------------------+
| Connector List                          | Connector Detail                       |
| - qodercli      L3  Healthy             | Name: qodercli                         |
| - codex-desktop L2  Warning             | Type: Mature CLI                       |
| - github-mcp    L3  Healthy             | Auth: Connected                        |
| - custom-skill  L2  Healthy             | Health Check: PASS                     |
|                                         | Policies: ManualOnly=Off               |
|                                         | Mode: Interactive PTY / Managed        |
|                                         | [测试连接] [禁用] [查看错误日志]        |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- 一级 Tab：
  - `Skills`
  - `MCP`
  - `Desktop Clients`
  - `Mature CLI`
- 每个 Tab 包含：连接列表、能力等级、健康状态、最近错误、测试入口

#### 关键交互

- 连接器统一流程：`创建 -> 授权/路径配置 -> 健康检查 -> 权限确认 -> 启用`
- 每个连接器显示能力等级 `L1/L2/L3`
- 支持“一键禁用”“仅手动调用”“仅沙箱可用”等策略开关

#### 分类差异（必须明确）

- `Skills`：偏流程能力与知识封装，重点展示版本与依赖
- `MCP`：偏工具目录，重点展示工具数量、标签、鉴权状态
- `Desktop Clients`：偏桌面应用控制，重点展示接入方式（API/Bridge/GUI）
- `Mature CLI`：偏终端原生交互，重点展示 PTY 兼容性与审计策略

### 4.4 Workspace（工作空间中心）

#### 页面目标

把“路径边界 + 执行边界 + 风险边界”可视化，减少误操作焦虑。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| Workspace                                                                        |
+----------------------------------------------------------------------------------+
| Workspace List            | Scope & Paths                  | Policies            |
| - /proj/app (Trusted)     | Root: /Users/me/proj/app      | Cmd Allowlist       |
| - /proj/data              | Allowed: src/, tests/, docs/  | Network: sandbox-only|
| - /sandbox/tmp            | Denied: ~/.ssh, ~/Downloads   | Approval: push/delete|
|                           | [路径测试 Dry-run]             | Env Allowlist        |
+---------------------------+--------------------------------+---------------------+
| Ephemeral Workspace: [ON]  High-Risk tasks run in sandbox clone                 |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- 左侧：Workspace 列表（最近、收藏、Trusted）
- 中间：目录范围与可访问策略
- 右侧：命令权限、网络权限、审批规则、环境变量策略
- 底部：测试区（策略 dry-run，不真实执行）

#### 关键交互

- 新建 workspace 必须完成路径扫描与权限提示
- 支持 `Trusted Workspace` 标记，但高风险动作仍需审批
- 支持 `Ephemeral Workspace` 开关（默认高风险任务走沙箱副本）

### 4.5 CLI Studio（CTS 自定义 CLI）

#### 页面目标

让用户将高频任务沉淀为可复用命令，形成个人/团队资产。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| CLI Studio                                                                       |
+----------------------------------------------------------------------------------+
| Command Tree               | Command Editor                 | Release / Policy    |
| - proj.fix-tests           | Name: proj.fix-tests           | Scope: Team         |
| - proj.bump-deps           | Params: repo, branch, scope    | Approval: required  |
| - ops.pre-release-check    | Steps: lint -> test -> report  | Version: v0.3.1     |
|                            | Output Map: summary,diff       | [Dry-run] [Publish] |
+----------------------------+---------------------------------+---------------------+
| Footer: Source = "Generate from Task Replay" [导入最近任务]                       |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- 左侧：命令目录（草稿/已发布/已废弃）
- 中间：命令编辑器（参数、步骤、条件、输出映射）
- 右侧：策略面板（权限、审批、可见范围、版本说明）
- 底部：验证与发布（dry-run、测试、发布、回滚）

#### 关键交互

- 支持“从任务回放生成命令”一键预填
- 发布前强制执行校验：参数完整性、策略完整性、回滚路径
- 发布后进入命令目录与运行面板，可在任务中直接作为 `cts_cli` 节点调用

### 4.6 Governance（团队治理）

#### 页面目标

提供企业必须能力：审计、权限、预算、合规导出。

#### 文字线框图（简版）

```text
+----------------------------------------------------------------------------------+
| Governance                                                                       |
+----------------------------------------------------------------------------------+
| Tabs: [Policy] [Audit] [Cost] [Access]                                           |
+----------------------------------------------------------------------------------+
| Policy Rules                              | Audit Stream                          |
| - block rm -rf outside workspace          | 10:11 userA approved git push        |
| - require approval for prod deploy        | 10:12 task#T-1 switched to sandbox   |
| - sandbox-only for high risk tasks        | 10:13 qodercli session resumed       |
|                                           | ...                                   |
+-------------------------------------------+---------------------------------------+
| Budget: this month 68% used | Alerts: 2 | Export: [CSV] [JSON]                   |
+----------------------------------------------------------------------------------+
```

#### 页面结构

- `Policy`：审批规则、风控策略、命令白黑名单
- `Audit`：任务轨迹、审批记录、会话证据
- `Cost`：模型消耗、沙箱成本、预算阈值
- `Access`：角色权限（Owner/Maintainer/Member/Viewer）

## 5. 跨能力统一交互规则

### 5.1 执行器选择规则

- `Auto` 模式默认按能力等级择优（L3 > L2 > L1）
- 用户可手动锁定执行器，锁定后不自动降级
- 若锁定执行器不可用，必须提示并请求用户确认降级

### 5.2 审批触发规则

统一审批入口，不因能力类型变化而改变交互：

- 触发高风险动作 -> `WaitingApproval`
- 显示动作摘要、风险原因、影响范围、建议选项
- 用户操作：批准 / 拒绝 / 改为沙箱执行 / 转人工接管

### 5.3 失败恢复规则

- 每个失败步骤统一提供恢复动作：
  - 原地重试
  - 切换执行器重试
  - 切换沙箱重试
  - 生成人工处理指南

## 6. 成熟 CLI 原生交互设计（PTY）

该部分用于区分“普通命令执行”与“终端原生操作”。

### 6.1 入口与模式

- 在 `Mission` 的执行配置中可选择：
  - `Managed`（结构化自动执行）
  - `Interactive PTY`（原生终端交互）
- 若任务模板标记“需要原生交互”，默认选 `Interactive PTY`

### 6.2 终端面板交互

- 主视图采用终端渲染组件，完整显示 ANSI/TUI
- 顶部显示会话状态、连接器名称、workspace、审计状态
- 提供快捷动作：发送 Ctrl+C、清屏、重连、导出日志

终端面板文字线框图：

```text
+----------------------------------------------------------------------------------+
| Interactive PTY: qodercli | Session: Active | Workspace: /proj/app | Audit: ON   |
| [Ctrl+C] [Clear] [Reconnect] [Export Log] [Switch Managed]                      |
+----------------------------------------------------------------------------------+
| qodercli> Select a plan:                                                         |
|   1) quick-fix                                                                   |
| > 2) safe-refactor                                                               |
|   3) deep-analysis                                                               |
|                                                                                  |
| [User presses Enter]                                                             |
| -> Running safe-refactor...                                                      |
| -> Step 2/6 complete                                                             |
+----------------------------------------------------------------------------------+
| Input status: passthrough | cols=160 rows=40 | TraceId=...                       |
+----------------------------------------------------------------------------------+
```

### 6.3 选择与回车场景（如 qodercli）

- 菜单选择事件实时渲染，不做中间层文本重排
- 用户按键（方向键/数字键/回车）原样透传到 PTY
- 回车后若命中高风险命令，先转审批再继续

### 6.4 会话断开与重连

- 前端关闭不立即销毁 PTY 会话（进入 Detached）
- 用户重开 Runbook 可“重新附着”继续操作
- 会话恢复后显示“断连期间输出摘要”

## 7. 能力区分矩阵（界面展示建议）

为了让用户知道“该用哪类能力”，建议在 `Integrations` 和 `Mission` 都显示以下维度：

- 交互类型：自动化 / 原生交互 / 半自动
- 稳定性等级：L1/L2/L3
- 审批强度：低/中/高
- 可回放性：完整/部分/弱
- 推荐场景：探索、批处理、生产变更、故障排查

## 8. 首次使用引导（Onboarding）

三步引导，避免功能过多造成劝退：

1. 选择 Workspace（必须）
2. 接入一个成熟 CLI（推荐 qodercli）与一个 MCP
3. 运行“Hello Workflow”模板（含一次审批演示）

引导完成后自动生成：

- 一个可复用模板
- 一条 CTS 示例命令
- 一份任务回放报告

## 9. 页面级文案与反馈规范

- 主按钮文案用动作词：`运行任务`、`开始规划`、`请求审批`
- 错误提示要可执行：给出原因 + 修复按钮
- 风险提示要具体：说明影响对象与可逆性
- 所有关键动作反馈不超过 1 秒出现状态变化

## 10. V0 实施建议（界面范围）

V0 优先页面：

- Mission
- Runbook
- Integrations（仅 Mature CLI + MCP）
- Workspace（基础策略）

V0 可延后页面：

- CLI Studio 高级编排器（先保留基础命令创建）
- Governance 成本中心高级报表

V0 交互验收：

- 新用户 10 分钟内完成首个任务
- 用户能独立完成一次审批与恢复
- 用户能在 2 分钟内定位失败步骤并选择恢复策略

## 11. 多执行器协作任务设计（客户端、CLI、MCP、Skill 联动）

本节定义“一个任务需要多个能力协作”的标准交互与执行模型，覆盖桌面客户端、成熟 CLI、MCP 与 Skill 的混合编排。

### 11.1 任务模型：从线性步骤升级为任务图（DAG）

每个任务不再是单链路，而是由多个可并行/串行的节点组成。

节点字段建议：

- `node_id`：步骤唯一标识
- `executor_type`：`desktop` / `mature_cli` / `mcp` / `skill` / `cts_cli`
- `connector_id`：具体连接器
- `mode`：`managed` / `interactive_pty`
- `depends_on`：依赖节点列表
- `input_contract` / `output_contract`：输入输出结构
- `workspace_binding`：绑定的 workspace
- `risk_level`：`low` / `medium` / `high`

图级别字段建议：

- `task_id`、`trace_id`
- `goal`（用户目标）
- `routing_policy`（自动路由策略）
- `fallback_policy`（失败回退策略）

### 11.2 规划与路由流程（Planner + Router）

任务提交后执行以下流程：

1. Planner 生成候选任务图（步骤、依赖、并行关系）
2. Router 为每个节点选择执行器，默认优先级：`L3 > L2 > L1`
3. PolicyGate 校验 workspace 与风险策略
4. 生成“执行前预览”：展示每个节点将调用什么能力
5. 用户确认后进入 Runbook 执行

交互规则：

- 用户可以锁定某些节点执行器（如强制 qodercli）
- 锁定执行器不可用时必须弹出降级确认
- 禁止静默降级到 GUI 自动化

### 11.3 跨能力交接包（Handoff Packet）

为避免“上一步结果下步不可用”，每个节点结束后必须输出标准交接包。

交接包内容建议：

- `producer_node`：产出节点
- `payload_json`：结构化结果
- `artifacts`：文件、diff、报告、截图等
- `summary`：给下一节点的人类可读摘要
- `evidence`：日志片段、trace、校验结果
- `sensitivity`：是否含敏感内容（用于脱敏与权限控制）

交互呈现：

- Runbook 中可点击“查看上游输入来源”
- 节点详情显示“输入来自哪个节点、何时生成、是否已验证”

### 11.4 Runbook 多泳道视图（核心交互）

当任务包含多类能力时，Runbook 默认切换为“泳道模式”。

泳道建议：

- `MCP/Skill Lane`
- `CLI Lane`
- `Desktop Lane`
- `CTS Lane`

文字线框图（多执行器协作）：

### 11.5 审批与暂停策略（跨执行器统一）

审批按“节点”触发，但在“任务级队列”统一展示。

规则：

- 任一节点触发高风险动作 -> 该节点进入 `WaitingApproval`
- 默认仅暂停相关分支；若为全局高风险动作可暂停全任务
- 审批结果可选：
  - 批准继续
  - 拒绝并中断节点
  - 改为沙箱执行后重试
  - 人工接管该节点

可视化要求：

- 审批卡显示：来源节点、上游输入、影响范围、可回滚性
- 审批后自动写入审计流并广播给关联节点

### 11.6 失败恢复与局部重跑

多执行器任务必须支持“局部恢复”，避免整任务重跑。

恢复策略：

- `Retry Node`：原执行器原参数重试
- `Retry With Another Executor`：切换执行器（如 qodercli -> codex）
- `Retry In Sandbox`：复制上下文到沙箱后重跑节点
- `Manual Takeover`：人工执行后回填结果，继续 DAG

交互要求：

- 节点失败后显示推荐恢复路径（基于失败类型）
- 允许“从该节点及其下游重跑”
- 保留成功节点快照，避免重复消耗

### 11.7 典型协作模板（可直接落地）

模板：`故障修复并提交 PR`

1. `N1(MCP)`：拉取最近 1h 错误日志  
2. `N2(Skill)`：生成根因假设与修复建议  
3. `N3(Mature CLI-PTY)`：qodercli 按建议修改代码  
4. `N4(CTS CLI)`：执行 `pre-release-check`  
5. `N5(Desktop Client)`：生成 PR 描述与变更说明  
6. `N6(Mature CLI-PTY)`：执行 `git push`（触发审批）  
7. `N7(Desktop/MCP)`：创建 PR 并回写链接

模板交互特性：

- N3/N6 保持原生终端交互
- N4 结构化输出供 N5 使用
- N6 审批后继续，不影响已完成节点

### 11.8 客户端实现边界（V0->V1）

V0：

- 支持串行 + 小规模并行（最多 2 条并行分支）
- 支持节点级审批与节点级重试
- 支持单 workspace 下的多执行器协作

V1：

- 支持复杂 DAG（多分支汇合）
- 支持跨 workspace 的受控协作（需显式授权）
- 支持任务图模板化与参数化复用
